> 版本：v1.0（2026-06）  
> 适用范围：dingma 小程序文案智能体  
> 关联方案：[`docs/plans/dingma-知识库方案.md`](../plans/dingma-知识库方案.md)

## 一、本次更新概述

### 1.1 解决什么问题

学员使用顶妈文案智能体生成朋友圈、种草、新品预热等文案时，模型可能**编造未列出的配料、功效或工艺**。本次更新通过 **MySQL 产品知识库 + 对话前自动注入产品事实**，约束生成内容「只讲有的、不讲没有的」。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **存全量、用分层** | 配方（克重、步骤）全量入库；文案智能体只用轻量 `copywriting_facts` |
| **dingma 隔离** | 独立 `/dingma/chat` 接口与代码目录，**不修改主程序** `/chat` |
| **不选产品** | 学员在指令中写明产品名，系统模糊匹配；未写明则注入产品索引并引导补充 |
| **一次灌库、长期复用** | PDF 课件数据一次性 seed；未来售后智能体读同表，无需重导 |

### 1.3 变更范围一览

| 模块 | 变更 | 说明 |
|------|------|------|
| dingma 小程序 | 改 1 处 API | `generate.ts` → `/api/v1/client/dingma/chat` |
| 主程序 / 火源小程序 | **无变更** | 仍走 `/api/v1/client/chat` |
| 后端主程序 `creation.py` | **无变更** | 逻辑 fork 到 dingma 目录 |
| MySQL | 新表 | `dingma_product_knowledge`（101 条，租户 dingma） |
| 管理后台 | 本期无 CRUD 页 | 数据通过 JSON + seed 脚本维护 |

## 二、架构说明

### 2.1 请求链路

```
学员选智能体 → 输入含产品名的指令
    ↓
dingma 小程序 POST /api/v1/client/dingma/chat
    ↓
chat_executor（技能组装 + IP 人设）
    ↓
knowledge.resolve（按产品名/别名匹配 → 注入 copywriting_facts）
    ↓
LLM 生成文案
```

### 2.2 代码目录（dingma 专用）

```
backend/
├── routers/client/dingma/
│   └── chat.py                 # 路由：POST /dingma/chat
├── services/dingma/
│   ├── constants.py            # 铁律模板、注入模式枚举
│   ├── knowledge.py            # 匹配 + Prompt 块组装
│   └── chat_executor.py        # 对话执行（fork creation.py，2026-06-11）
├── models/dingma_product_knowledge.py
├── data/dingma/
│   └── product_knowledge_2026.json   # 全量产品数据（101 条）
├── db/migrations/
│   └── add_dingma_product_knowledge.py
└── scripts/
    ├── seed_dingma_product_knowledge.py
    ├── test_dingma_knowledge_unit.py
    └── test_dingma_chat.sh
```

### 2.3 数据表字段

表名：`dingma_product_knowledge`

| 字段 | 本期文案是否注入 | 用途 |
|------|-----------------|------|
| `product_name` / `aliases` | 用于匹配 | 用户输入模糊识别产品 |
| `copywriting_facts` | **是** | 含/不含/可写/不可写（无克重） |
| `pack_formula` | 否 | 出货配比（含克重），售后智能体预留 |
| `recipe_detail` | 否 | 制作步骤 JSON，售后智能体预留 |
| `source_version` | — | 课件版本，如 `2026-01` |

### 2.4 注入模式（预留扩展）

| 模式 | 常量 | 注入内容 | 使用方 |
|------|------|---------|--------|
| 文案 | `copywriting` | `copywriting_facts` + 铁律 | **当前全部 dingma 文案智能体** |
| 售后 | `after_sales` | `pack_formula` + `recipe_detail` | 未来「产品售后智能体」（未实现） |

## 三、部署与运维

### 3.1 新环境部署步骤

```bash
# 1. 数据库迁移
cd backend
python -m db.migrations.add_dingma_product_knowledge

# 2. 灌库（需 tenants 表存在 code=dingma 的租户）
python -m scripts.seed_dingma_product_knowledge

# 3. 单元测试（可选）
python -m scripts.test_dingma_knowledge_unit

# 4. 重启后端服务
```

### 3.2 课件更新后如何维护数据

1. 编辑 [`backend/data/dingma/product_knowledge_2026.json`](../../backend/data/dingma/product_knowledge_2026.json)（或复制为新版本文件如 `product_knowledge_2027.json` 并改脚本路径）
2. 每条记录必须包含唯一 `product_code`
3. 重新执行 seed（按 `tenant_id + product_code` 幂等 upsert）：

```bash
cd backend && python -m scripts.seed_dingma_product_knowledge
```

4. **无需改代码、无需重启**（数据热更新）；若改了 Python 逻辑则需重启后端

### 3.3 联调测试

```bash
# 需替换 TOKEN、AGENT_ID
export TOKEN=你的JWT
export AGENT_ID=智能体ID
bash backend/scripts/test_dingma_chat.sh
```

测试用例建议：
- 含产品名：「帮我写泡菜朝鲜面的朋友圈」→ 应命中单品事实
- 不含产品名：「帮我写早餐推广文案」→ 应返回产品索引 + 引导补充

## 四、运营与学员使用指引

### 4.1 学员侧（无需新 UI）

- 照常：**选智能体 → 输入指令**
- **必须在指令中写清产品名**（官方名称或常见别名），例如：
  - ✅ 「帮我写一段**泡菜朝鲜面**的朋友圈，突出私房早餐」
  - ✅ 「**番茄牛肉米线**新品预热文案，7 天节奏」
  - ⚠️ 「写个米线文案」→ 系统只能给品类级引导，不会编造具体配料

### 4.2 管理后台需配置（手动）

为 dingma 文案智能体建议：

1. **新建 `rule` 技能**（铁律），示例内容：

```
你只能使用系统提供的【产品事实】中的信息。
禁止添加未列出的食材、配料、功效、工艺、价格。
若用户未指明产品且知识库未匹配到，不要编造具体产品细节，应提示补充产品名。
```

2. 将该技能加入智能体 `skill_ids`
3. 设置 `is_routing_enabled = 0`（静态加载全部技能，避免向量误选）
4. 营销写法（朋友圈结构、新品上架节奏等）继续用 `model` 技能，与知识库解耦

### 4.3 别名维护建议

匹配依赖 `product_name` + `aliases`。常见需维护的别名示例：

| 产品名 | 建议别名 |
|--------|---------|
| 泡菜朝鲜面 | 朝鲜面、泡菜面 |
| 辣白菜荞麦面 | 荞麦面、辣白菜面 |
| 番茄牛肉米线 | 番茄米线、番茄面 |

新品上架时在 JSON 中补充 `aliases`，重跑 seed 即可。

## 五、后续扩展计划

### Phase 2：后台 CRUD（建议优先级：高）

| 项 | 说明 |
|----|------|
| 目标 | 运营无需改 JSON，后台直接改单品 |
| 路径 | `routers/admin/dingma/product_knowledge.py` + 前端管理页 |
| 字段 | 支持编辑全字段（含 `pack_formula`、`recipe_detail`） |

### Phase 2：产品售后智能体（建议优先级：中）

| 项 | 说明 |
|----|------|
| 目标 | 学员问制作步骤、克重、工艺时精确回答 |
| 实现 | 新建 `dingma/after_sales_chat.py`，`inject_mode=after_sales` |
| 数据 | **读同一张表**，无需重新导入配方 |
| 注意 | token 较大，需分段注入（成品配比 vs 制作详情） |

### Phase 2：生成后 audit（建议优先级：低）

| 项 | 说明 |
|----|------|
| 目标 | 检测生成稿是否出现 `不可写` 或未登记成分 |
| 实现 | 关键词/规则校验，命中则提示重写 |

### 明确不做（除非业务强需求）

- 小程序选品 UI（当前用「指令写清产品名」替代）
- 向量 RAG 检索配方（精确匹配优先，RAG 易错配）
- 修改主程序 `creation.py` 或 `/chat` 行为

## 六、注意事项与风险

### 6.1 隔离与回归

- **禁止**在 `creation.py` 中加入 dingma 知识库逻辑；所有 dingma 改动限定在 `routers/client/dingma/` 与 `services/dingma/`
- `chat_executor.py` 为 fork 副本，主程序 `creation.py` 升级时需**人工对比合并** bugfix，文件头已标注 fork 日期
- 火源主小程序、PC 端对话**不受影响**，回归时只需验证 `/chat` 仍正常

### 6.2 匹配与准确性

| 场景 | 行为 | 运营应对 |
|------|------|---------|
| 指令含明确产品名 | 注入该品 `copywriting_facts` | 引导学员用官方产品名 |
| 指令模糊 / 无产品名 | 注入全量产品名索引 + 铁律 | 话术模板要求写清产品 |
| 近义词冲突 | 取得分最高的一条 | 完善 `aliases`，避免过短别名 |
| 文案不要求克重 | 不注入 `pack_formula` | 符合预期；售后场景再用全量字段 |

### 6.3 数据一致性

- seed 脚本按 `product_code` upsert；**不要**随意改已有 `product_code`，否则会产生重复记录
- PDF 页间若有配比差异，以 JSON 中人工审定版本为准（单一事实源）
- `copywriting_facts` 应与 `pack_formula` 成分一致；改配方时两处同步更新

### 6.4 性能与 Token

- 命中 1 个产品：知识块约 300~500 字，token 压力小
- 未命中：注入产品索引约 2KB，仍可控
- dingma 专用截断策略：**优先保留知识块**，再截断后半段 agent 能力（见 `chat_executor.py`）

### 6.5 租户与环境

- 知识库数据绑定 `tenants.code = dingma`（当前库内 tenant_id=2）
- 新环境部署前确认 dingma 租户已创建，否则 seed 会失败
- 请求需带 `X-Tenant-Code: dingma`（小程序 `generate.ts` 已配置）
